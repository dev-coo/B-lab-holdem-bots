# B-lab Holdem Bots

B-lab 홀덤 봇 대회 참가자들이 각자 만든 봇과 그 **전략·프롬프트 노하우**를 모으는 공간입니다.

대회는 끝났고, 여기 모인 6명의 봇은 모두 **AI 코드 도구(Claude Code, Codex, Cursor 등)를 활용해서** 만들어졌습니다. 이 레포의 진짜 자료는 봇 코드 자체보다도, **각자가 그 AI를 어떻게 다뤘는지**에 있습니다.

## 왜 이 자료를 모으는가 (비교 축)

대회 결과(승·패 순위)는 모두가 압니다. 이 자료의 가치는 **그 결과에 도달하기까지 6명이 어떻게 달랐는가**를 비교할 수 있다는 데 있습니다. 각자의 README는 다음 세 축에서 비교 가능하도록 정리됩니다:

1. **지시 스타일 차이** — 결정적이었던 프롬프트의 형태, 명령형 vs 위임형, 검증을 어떻게 시켰는지
2. **키워드·도구·에이전트 차이** — 자주 등장한 호출 키워드, 슬래시 커맨드·MCP 도구·서브에이전트 사용 양상 (도구별로 가능한 수준에서)
3. **도메인 지식의 형태와 그 효과** — 초기 가설을 어디에서 가져왔고(도메인 지식/직관/데이터 분석/논문/모방 등) 그 가설이 끝까지 유효했는지, 중간에 폐기됐는지, 혹은 **본인의 도메인 지식이 오히려 가설을 너무 일찍 좁혀 발목을 잡았는지**

세 번째 축은 특히 흥미로운 가능성을 담고 있습니다 — 도메인 지식이 많은 사람은 "이렇게 해야 한다"는 사전 신념 때문에 AI에게 좁은 지시만 던지게 되고, 도메인 지식이 적은 사람은 "어떻게 이기지?" 같은 더 열린 질문으로 AI의 탐색 공간을 넓게 쓰는 경향이 있을 수 있습니다. 6개의 사례가 모이면 그 가설이 검증 가능해집니다.

## 폴더 구조

```
B-lab-holdem-bots/
├── README.md               # 이 파일
├── eunwoo/                 # 각 참가자의 닉네임 폴더
│   ├── README.md           # 본인 봇·전략·프롬프트 노하우 정리 (아래 표준 프롬프트로 생성)
│   ├── (봇 파일들)         # 1·2·3 라운드 출전 봇 코드
│   ├── lib/ 또는 의존성    # 독립 실행에 필요한 모듈
│   ├── requirements.txt    # 파이썬 의존성
│   └── run.sh              # 실행 스크립트
├── {닉네임2}/
└── ...
```

각 폴더는 **완전 독립 실행 가능**하도록 만듭니다. 다른 사람이 `cd {닉네임} && pip install -r requirements.txt && bash run.sh` 만으로 돌릴 수 있게.

본인이 만들지 않은 코드(레퍼런스로 받은 봇 등)가 폴더에 들어간다면 **반드시 출처 명시**하고 본인 것처럼 꾸미지 말 것.

## 사용 가이드 (자기 폴더 만드는 4단계)

각자는 **자기 봇 코드와 AI 세션 히스토리가 있는 환경**에서 진행합니다.

**1) (한 번만) 이 공유 레포를 로컬에 clone**
```bash
git clone https://github.com/dev-coo/B-lab-holdem-bots.git ~/path/to/B-lab-holdem-bots
```

**2) 본인 봇 프로젝트 디렉토리로 이동 → 표준 프롬프트 상단 입력값을 직접 채운 뒤 본인 AI에 복붙**
- **환경 3개** — 닉네임 · 공유 레포 로컬 경로 · 봇 프로젝트 위치
- **회고 5~6개** — 도메인 지식 2개 + 시행착오 3개 + (선택) 자유 회고 1개. **자동 추출 전에 본인 기억으로 먼저 적어 두기** — 한두 줄씩.
- AI가 본인 세션 + 봇 코드를 분석해 `{공유레포 경로}/{닉네임}/` 안에 봇 파일·의존성·`requirements.txt`·`run.sh`·`README.md`를 자동 생성. README의 **섹션 6**은 사용자가 미리 적은 회고, **섹션 7**은 AI의 "회고 ↔ 자료 검토".

**3) (선택) 섹션 7 검토 결과를 보고 섹션 6 회고를 정정 또는 유지**
- AI가 보고한 회고와 자료 사이 불일치를 검토. 기억과 데이터가 갈리는 지점 자체가 자료가 됨 — 정정해도 좋고 그대로 둬도 좋음.

**4) 공유 레포로 이동해서 commit & push**
```bash
cd ~/path/to/B-lab-holdem-bots
git add {닉네임}/
git commit -m "Add {닉네임} bot"
git push
```

> ChatGPT 웹처럼 **파일 시스템 접근이 없는 AI**를 쓰는 경우: AI가 모든 파일 내용을 메시지로 출력 → 본인이 수동으로 해당 경로에 저장.

````markdown
# 추출 항목

## [자동 추출 — 세션·코드 분석 기반. 사용자에게 묻지 말고 직접 추출]

### 1. 봇 소개
한 줄 설명 + 어떤 컨셉·접근으로 만든 봇인지. **정해진 분류(TAG/LAG/GTO 등)에 억지로 끼워 맞추지 말 것.** 사용자가 세션에서 봇을 어떻게 정의·표현했는지 그 어휘 그대로 가져오기.

### 2. 핵심 전략
의사결정 로직을 2~5문단으로. 어떤 시그널(스택, 포지션, 상대 액션, 보드 텍스처, 확률 모델, 추정값 등)을 보고 어떻게 결정하는지. 코드와 세션 양쪽에서 추출.

### 3. 사용한 AI 도구 · 개발 과정
- 어떤 AI 도구(들)를 썼는지, 대략 몇 세션 정도 진행했는지
- **라운드별 봇 교체 이력** — 1·2·3 라운드 각각 어떤 봇으로 출전했고, 왜 바꿨는지 (세션 텍스트·커밋 메시지에서 추출)

#### 3-A. 도구·에이전트 사용 양상 (정량, 가능한 수준에서)
세션 로그에서 직접 카운트해서 채우세요. 비-Claude Code 사용자는 본인 도구의 동등 항목으로 대체하거나 "해당 없음".

- **슬래시 커맨드 사용 빈도 (top 5~10)** — 각 커맨드 이름과 호출 횟수. 사용 안 했으면 그렇게 쓰기.
- **MCP 도구 호출 양상** — 어떤 MCP 도구를 어떤 용도로 썼는지. 빈도까지 정확할 필요는 없고 어떤 종류·용도였는지.
- **서브에이전트·역할 사용** — 어떤 종류의 에이전트를 어떤 단계·작업에 썼는지. 빈도 + 무슨 작업이었는지 한 줄.
- **자동화 vs 대화 비중** — 슬래시 커맨드/스킬로 일을 던지는 비중이 큰지, 직접 대화로 한 줄씩 진행하는 비중이 큰지 대략의 인상.

### 4. 프롬프트 스타일 (세션 텍스트에서 직접 추출)
사용자에게 묻지 말고 세션 로그를 직접 읽어서 다음을 채우세요:

- **(a) 메타 규칙** — 세션에서 반복적으로 등장한 워크플로 패턴. 실제 빈도로 판단.
- **(b) 자주 쓰는 표현 패턴** — 사용자가 반복해서 쓴 한두 개 문장·말투·지시 형태.
- **(c) 결정적이었던 프롬프트 1~3개** — 임팩트가 컸던(=직후 코드 변화가 컸던) 프롬프트를 **세션에서 원문 그대로 발췌**해 코드블록으로 인용. 오타·반말·비효율적 표현 그대로.
- **(d) 자주 등장한 키워드 (top 10 정도)** — 사용자 발화에서 반복 호명된 도메인·검증·결정 어휘. 단순 빈출어가 아니라 **본인의 사고를 호명한 키워드**여야 함. 가능하면 도메인 어휘(전문용어) vs 메타 어휘(분석·전략·패턴) 비중도 한 줄.
- **(e) 초기 가설 + 프롬프트 톤** — 두 부분:
  - **초기 가설** — 첫 5~10 세션의 사용자 발화에서 봇 방향 선언문 1~3개를 원문 그대로 인용. 그 가설이 끝까지 유효했는지, 중간에 폐기됐는지 한 줄 첨부.
  - **프롬프트 톤** — 명령형 비중 vs 위임·질문형 비중을 대략 비율로. 결정의 무게중심이 본인에게 있었는지 AI에게 있었는지.

다섯 종류 다 못 찾으면 해당 항목만 "패턴이 안 잡힘 — 즉흥적 사용 위주"로 솔직히 적기.

### 5. 돌리는 방법
코드(특히 `run.sh`, `requirements.txt`, 봇 파일 헤드)를 보고 추출. 실행 커맨드, 의존성, 환경 변수, 인자.

# 작성 시 주의사항

1. **코드 양보다 "어떤 사고·어떤 프롬프트 다룸이었는지"가 핵심.** 6명을 비교해서 읽었을 때 인사이트가 나오는 게 목적.
2. **안 쓴 항목은 비워두지 말고** "해당 없음"이라도 명시.
3. **프롬프트 인용은 가공 금지** — 오타, 반말, 비효율적 표현이 있어도 그대로. 그게 진짜 자료.
4. **순서를 지킬 것** — 자동 추출(섹션 1~5) → 입력에서 받은 사용자 회고를 섹션 6에 배치 → 회고 ↔ 자료 검토를 섹션 7로 작성 → 사용자에게 정정 또는 유지 안내.

# 출력

다음 파일들을 `{공유레포 경로}/{닉네임}/` 폴더 아래에 생성:

1. `README.md`
   - **섹션 1~5 (자동 추출)** — 세션·코드 분석으로 채운다.
   - **섹션 6 (시행착오 · 회고)** — 입력에서 받은 사용자 회고(도메인 지식 2 + 시행착오 3 + 선택 자유 회고)를 그대로 배치. 가공·요약하지 말 것.
   - **섹션 7 (회고 검토)** — 다음 세 가지를 비판적이되 단정하지 말고 보고:
     1. **회고 ↔ 자료 불일치** — 회고가 자료 안 데이터와 충돌하는 항목
     2. **회고가 추측으로 남긴 항목 중 자료가 답을 주는 경우** — 단정 가능한 부분을 인용으로 제시
     3. **회고에서 빠진 큰 사건** — 자동 추출에서 발견했지만 회고에 없는 굵직한 사건
2. 1·2·3 라운드에 출전한 봇 파일들 (세션·커밋 분석으로 확정)
3. 그 봇들이 의존하는 lib·헬퍼 모듈 카피 (폴더가 독립적으로 실행 가능하게)
4. `requirements.txt` (실제 사용된 파이썬 의존성)
5. `run.sh` (실행 스크립트 — 봇 어떻게 띄우는지)

마지막에 사용자에게 "섹션 7 검토 결과를 보고 섹션 6 회고를 정정할지 그대로 둘지 결정한 뒤 공유 레포에서 commit & push 하세요"라고 안내한다.
````
